# Open Questions

### 1. Action Return Values

Some actions naturally return data to the caller (e.g., a `read` action should
return a value). Proposed shape:

```mc
let (reader, count) = reader.read()?;
```

The session method returns a tuple of the next session state and the result
value. The action handler returns a tuple:

```mc
action read(counter: Active) {
    (counter, counter.count)
}
```

### 2. Multi-Type Hosting

Can a machine host more than one linear type?

```mc
machine DevService hosts PullRequest(key: id), Issue(key: id) { ... }
```

Recommendation: defer to V2. Single-type hosting covers most use cases.

### 3. Instance Lifecycle Hooks

Should the machine receive callbacks when instances reach `@final` or when
sessions disconnect?

Recommendation: yes. Optional hooks:
- `fn on_final(key: KeyType, state: ...)` — called when instance reaches
  `@final`.
- `fn on_session_close(session_id: u64)` — called when a session ends.

Default behavior: auto-remove instance on `@final`, no-op on session close.

### 4. Persistence Integration API

The exact API for connecting a machine to a storage backend is open. The
architectural commitment is that persistence is the machine's responsibility
and is invisible to session callers. The identity key field is the persistence
key.

### 5. `nowait` Non-Blocking Form

The exact syntax and type shape for non-blocking session steps is open.

```mc
let pending_op = nowait author.submit();
let pending = pending_op.wait()?;
```

The commitment is: blocking is the default, non-blocking is explicit, and the
result remains typed.

### 6. Identity Key Assignment

Who assigns the identity key — the machine or the caller?

- **Machine-assigned** (recommended default): `create` returns a session whose
  key was generated by the machine. The caller reads it via `session.id`.
- **Caller-assigned**: `create(PullRequest as Author, my_key)?` — the caller
  provides the key. Useful when the key comes from an external system (e.g.,
  a database-generated ID).

Both forms can coexist. The type declares the field; the machine decides the
assignment strategy.

### 7. Composite Identity Keys

Should composite keys be supported directly?

Recommendation: V1 supports single-field keys only. Composite keys can be
modeled as a single struct type:

```mc
type LineItemKey = { order_id: u64, line_number: u64 }

@linear
type LineItem = {
    id: LineItemKey,
    // ...
}

machine OrderService hosts LineItem(key: id) { ... }
```

### 8. Emitted Value Routing Configuration

How exactly emitted values are routed to channels is open. Options:
- Implicit: all emitted values from a hosted type automatically flow to
  co-hosted channels.
- Explicit: the machine declares which types route to which channels.
- Filtered: channels declare which types they accept.

### 9. `session` Context in Handlers

Handlers may need access to session metadata (e.g., `session.user` in the
`emit CommentAdded` example). The exact mechanism for accessing session context
inside handlers is open.

### 10. Multi-State Actions (Union States)

Some actions naturally apply to many source states — e.g., `comment` on both
Draft and Review, or `cancel` from any non-final state. In V1, each
source state requires a separate declaration and a separate method
implementation, which is verbose.

A possible V2 extension is inline state unions in action declarations:

```mc
actions {
    comment(text: string): (Draft | Review) -> same,
    cancel: (Draft | Review | Approved) -> Cancelled,
}
```

With corresponding methods using union receiver types:

```mc
fn comment(self: Draft | Review, text: string) -> same {
    self   // flat field work only, concrete state preserved
}

fn cancel(self: Draft | Review | Approved) -> Cancelled {
    Cancelled {}
}
```

This covers two common patterns:
- **Cross-cutting behavior**: many sources, target = source (`same` keyword).
- **Cross-cutting exit**: many sources, one concrete target.

Challenges:
- `same` is a new transition semantic: "output must be the same variant as
  input." The compiler must enforce this.
- Union-typed receivers cannot access state-specific fields — only flat
  (shared) fields. This limits union actions to cross-cutting concerns.
- Named state groups (e.g., `Commentable = Draft | Review`) could be
  introduced as aliases, but it's unclear whether they belong in the `states`
  block or as a separate concept.

Recommendation: defer to V2. V1 uses explicit per-state declarations and
methods with annotated receiver types (`self: Draft`). Union states can layer
on later as sugar over the same model.

### 11. Optional Method Blocks for Hosted-Only Types

When a linear type is only used in hosted mode and every action needs
infrastructure access, requiring base method implementations can feel like
boilerplate — trivial `fn action(self) -> State { self }` methods that nobody
calls directly.

Options:
- **Always require method blocks** — uniform model, small tax for consistency.
- **Method blocks optional for types with roles** — if no method block exists,
  the machine must provide all action handlers (no base to override). If a
  method block exists, the machine overrides selectively.

Recommendation: open. The second option is more pragmatic but adds a
conditional rule to the model.
